# The Foundry — Output Locations Protocol

**Purpose:** Every pipeline must document where its outputs land for the human. Code lives in git. But reports, briefs, presentations, and exports live in the user's file system — and that must be documented per-project.

## The Principle

The pipeline produces two categories of output:

1. **Code artifacts** — live in git repos, deployed via CI/CD. The Foundry handles this.
2. **Human artifacts** — reports, briefs, presentations, images, exports. These are what the human SEES and USES. The Foundry does NOT handle this — but it must know where they go.

If a pipeline stage generates a human artifact (a PDF, a deck, a report, a brief), it MUST know the output location. "I generated a report" without "and it's at /path/to/report.pdf" is useless.

## Per-Project Output Map

Every project that uses The Foundry MUST have an `OUTPUT-MAP.md` in its `.claude/` directory (or project root) that defines where human artifacts land.

### Format

```markdown
# Output Map — [Project Name]

## Output Root
[Base path where all project outputs live]

## Locations

| Output Type | Location | Generated By | Format |
|-------------|----------|--------------|--------|
| [type] | [path relative to output root] | [skill/stage] | [PDF/PNG/etc] |
```

### Example: LifeModo


## Rules

1. **Every project gets an OUTPUT-MAP.md** — even if the project only generates code. The map may be simple ("all outputs are in git").
2. **Gamma decks** always go to `~/decks/gamma-pdfs/{Project}/{Milestone}/` per the milestones-mirror-folders rule.
3. **iCloud paths** use the macOS-canonical format: `~/Library/Mobile Documents/com~apple~CloudDocs/Documents/`
4. **The Foundry stages** that generate human artifacts MUST log the output path to progress.txt:
   ```
   - OUTPUT: [artifact type] → [full path]
   ```
5. **The follow-up stage** checks that all OUTPUT entries in progress.txt actually exist on disk.

## Integration with The Foundry Pipeline

The `init_pipeline()` function in `foundry.sh` reads the project's OUTPUT-MAP.md (if it exists) and makes output paths available as environment variables for stages that need them.

This is NOT enforced in v2.0 — it's a documented convention. Enforcement comes in v2.1 when we have enough projects using it to validate the pattern.

## Why This Exists

Generated artifacts are ephemeral files in folders. Without documenting where they go:
- Next session can't find last session's outputs
- No audit trail of what was delivered
- Client asks "where's that report?" and nobody knows
- iCloud sync issues go undetected because nobody checked the path

The OUTPUT-MAP.md is the artifact's address book.
